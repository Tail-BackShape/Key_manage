# スマートロック状態管理・通知システム 仕様書 (v3.1)

## 1. プロジェクト概要
Raspberry Pi上のローカルサーバーを介し、ブラウザから施錠状態の記録と管理を行うシステム。操作ログはDiscordのWebhookを用いて指定チャンネルへリアルタイム通知する。

**【重要事項：ハードウェア制御について】**
実際の物理的な鍵の開閉は人間が手動で行う。そのため、**Raspberry PiのGPIOを用いたサーボモーターや電子錠などのハードウェア制御は一切行わない。** 本システムは純粋な「状態フラグのトラッキング」と「操作ログの記録・通知」のみを担当する。

## 2. システム要件
* **動作環境**: Raspberry Pi (単なるWebサーバーとして利用)
* **バックエンド**: Python (Flask または FastAPI)
* **フロントエンド**: HTML / CSS / JavaScript (動的DOM操作)
* **UI要件**: 7インチ程度のタッチディスプレイ運用を前提とし、操作ボタンは大きめに設計する（目安: 高さ80px以上、十分な余白）
* **外部連携**: Discord Webhook API
* **設定管理**: `.env` ファイルによるWebhook URLやユーザー名の管理

## 3. 通知フォーマット
操作が行われた際、以下の形式でDiscordへ送信する。
> **[時刻] [操作者]が [アクション] しました**
> * ※アクション例：開錠、一時施錠、施錠・帰宅

## 4. 状態定義と画面遷移

**UI方針（簡素化）**
* 1画面1目的を徹底し、情報量を最小化する。
* 補助テキストは短くし、主要操作ボタンを目立たせる。
* タッチ操作の誤タップ防止のため、ボタン間隔を広めに確保する。

### Phase 1：初期画面 (Index)
* **状態**: システムが「通常(LOCKED)」か「一時施錠(TEMP_LOCKED)」かを保持。
* **UI要素**:
  * `開錠`ボタン：押下で **Phase 2** へ。
  * `帰宅`ボタン：※状態が **TEMP_LOCKED** の時のみ表示。押下で **Phase 2** へ。

### Phase 2：ユーザー選択 (User Selection)
* **UI要素**: 設定ファイル（.env等）で定義されたユーザー名の一覧をボタンとして表示。
* **アクション**: ユーザー選択後、その情報を保持して **Phase 3** へ。

### Phase 3：操作選択 (Action Selection)
* **UI要素**: `帰宅`ボタン、`一時施錠`ボタン。
* **アクションと通信**:
  1. サーバーへ「ユーザー名」「選択アクション」を送信。
  2. サーバー側で時刻を取得し、Discordへ通知を送信。
  3. 「施錠・帰宅（home）」が選ばれた場合、状態を **LOCKED** に更新し、**Phase 1（開錠ボタンのみ表示）** へ遷移する。
  4. 「一時施錠（temp_lock）」が選ばれた場合、状態を **TEMP_LOCKED** に更新し、**Phase 2** へ戻る。

## 5. Codexへの実装指示（制約事項）
1. **GPIOライブラリのインポート禁止**: `RPi.GPIO` や `gpiozero` 等のハードウェア制御用コードは含めないこと。
2. **純粋なWebアプリとして実装**: アクション（開錠・施錠など）を受け取ったら、単に内部の変数（State）を更新し、DiscordへHTTPリクエスト（POST）を送るだけのロジックにすること。
3. **Notification Module**: `requests` 等を用いてDiscordのWebhookへJSONペイロードを送信する処理を実装すること。時刻はフォーマット（例:`%Y/%m/%d %H:%M:%S`）して含める。
